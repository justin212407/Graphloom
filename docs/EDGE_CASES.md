# GraphLoom — Edge Cases

Scenarios the implementation needs an explicit answer for, so they get decided here rather than improvised mid-code.

## Graph-side

**Cyclic graphs.** A→B→A is not a valid pipeline (no execution order exists). `graphToCode` should reject at validation time with a clear error, not attempt a partial topological sort.

**Orphan node.** A node with no edges in or out (other than `InputNode`/`OutputNode`, which are allowed to be endpoints). Valid but generates dead code — `graphToCode` should still emit it, since deleting a user's node without consent is worse than emitting unused code.

**Multiple edges into one input port.** Not allowed — an input port accepts exactly one incoming edge. Reject at the edge-creation step in the adapter, not deep in codegen.

**Multiple `OutputNode`s.** Rejected at parse/validation time (see `PARSER_RULES.md` §2, `NODE_TYPES.md` §4) — v1 graphs have exactly one terminal.

**Node deleted while its code was mid-edit.** This is a true conflict — falls under `CONFLICT_DETECTION.md`'s `"conflict"` status, with `keep: "code"` effectively re-adding the node and `keep: "graph"` discarding the in-progress code edit. No silent auto-resolution.

## Code-side

**Mid-edit invalid syntax.** While the user is actively typing in Monaco, the code is transiently unparseable on almost every keystroke. `codeToGraph` must never be called on a raw keystroke — debounce, and additionally attempt a parse-validity check before invoking the graph update. On parse failure, leave the canvas exactly as it was rather than clearing it or throwing a visible error mid-typing.

**Untagged functions mixed in with tagged ones.** Treated as passthrough (see `PARSER_RULES.md` §5) — preserved verbatim in the file, never surfaced as a node, never deleted.

**Renamed function symbol.** If a user renames `fetchResults` to `fetchSearchResults` by hand, the `@graphloom:node` tag is still present, but the `mapping.astToNode` lookup (keyed by prior symbol) will miss. Treat this as: the old node id is now orphaned in the mapping (drop it), and a new node is detected. This will read as a delete+add rather than a rename in v1 — acceptable for the 4-day scope, but worth stating explicitly rather than pretending renames are tracked.

**Destructured parameters / spread args.** Outside what the parser resolves (see `PARSER_RULES.md` §5) — the specific tagged function downgrades to passthrough with a warning, rest of the file parses normally.

**Untyped fetch return value.** No explicit return type on an `async function` tagged `fetch` → output port defaults to `"any"`, non-blocking warning attached (see `PARSER_RULES.md` §4). Does not fail the parse.

## Sync-layer

**Regenerating with no actual delta.** Must be a no-op for drift purposes — see `CONFLICT_DETECTION.md` §4. Guard against `graphToCode` producing spurious diffs (e.g. import ordering, trailing whitespace) that would falsely trigger `"graph-ahead"` on every sync.

**Empty graph.** Zero nodes is valid — `graphToCode` emits an empty (but well-formed) file with just the passthrough region, if any existed from a prior snapshot.

**First sync, no `prevSnapshot`.** Every code path in `graphToCode`/`codeToGraph` that reads `prevSnapshot.mapping` must handle `undefined` — this is the fresh-generation case from `IMPLEMENTATION_PLAN.md` Day 1, and it's also what happens if a user pastes in hand-written GraphLoom-tagged code that was never generated by this tool.

## Explicitly out of scope for v1 (documented, not silently ignored)

- Multi-file graphs (see `PARSER_RULES.md` §5)
- True rename tracking across the code→graph direction
- Automatic conflict resolution / three-way merge (see `CONFLICT_DETECTION.md` §6)
- Any node kind introducing control flow (branches, loops as graph structure — see `NODE_TYPES.md` §5)
